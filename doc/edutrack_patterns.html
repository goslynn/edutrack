BLOQUE 01
Autenticación y Permisos
🔐 AUTH Service
BE-AUTH-001BE-AUTH-002 BE-AUTH-004BE-AUTH-005 BE-AUTH-006BE-AUTH-007
JWT Stateless Auth Unix Permission Model RBAC Dinámico

El Auth Service combina autenticación stateless con JWT RS256 (par de claves asimétrico), roles completamente dinámicos con seed data mutable, y un modelo de permisos Unix-style donde cada par (role_uuid, resource_uuid) tiene un flag numérico r=4 | w=2 | x=1. El permiso efectivo de un usuario es la unión OR a nivel de bits de los flags de todos sus roles.

Por qué estos patrones

JWT con RS256 permite que cualquier microservicio valide el token usando solo la clave pública — sin llamar al Auth Service en cada request. El modelo Unix de flags por recurso mantiene al Auth Service como infraestructura pura: no conoce la semántica de cada recurso, solo gestiona el número. El dueño del recurso (ej: Course Service) registra sus resource_uuid y decide qué flags tienen significado.

BLOQUE 02
Permisos Granulares por Curso
📚 CRS Service
BE-CRS-002BE-CRS-003
Strategy Decorator — CDI Interceptor

El Course Service gestiona permisos contextuales por instancia de recurso: AUTH define permisos por tipo de recurso abstracto; CRS define si el docente X tiene acceso al curso concreto Y. Una tabla course_access(user_id, course_id, access_level) almacena las asignaciones. Un interceptor CDI —implementado como Decorator— evalúa el acceso antes de ejecutar cualquier lógica de negocio.

Por qué estos patrones

El Strategy permite definir dos comportamientos (READ vs WRITE) sin condicionales dispersos en el código. El Decorator en Quarkus se implementa con interceptores CDI: anotando el método con @CourseAccess(level=WRITE), el interceptor evalúa la tabla de asignaciones antes de continuar, manteniendo la lógica de negocio completamente limpia.

BLOQUE 03
API Gateway y Comunicación
🔀 API Gateway ⚙️ Transversal
BE-GW-001BE-GW-002 BE-GW-003BE-GW-004 BE-TRV-004BE-TRV-005
API Gateway Chain of Responsibility Correlation ID

El gateway procesa cada request a través de una cadena de responsabilidades: validación JWT → rate limiting → inyección de Correlation ID → enrutamiento. Cada eslabón puede cortar el flujo (401/429) o pasarlo al siguiente. El Correlation ID (UUID generado en el gateway) se propaga como header interno X-Correlation-Id a todos los microservicios, habilitando trazabilidad completa en CloudWatch y Jaeger.

Por qué estos patrones

La Chain of Responsibility permite agregar o reordenar eslabones sin modificar el resto del pipeline. En Quarkus se implementa con filtros JAX-RS (@Provider + ContainerRequestFilter) ordenados por prioridad anotada. El Correlation ID es fundamental en arquitecturas distribuidas: sin él, diagnosticar un error que cruza 3 microservicios es inviable.

BLOQUE 04
Resiliencia y Tolerancia a Fallos
🔔 NOT Service ⚙️ Transversal
BE-NOT-004BE-TRV-001 BE-TRV-002BE-TRV-003
Circuit Breaker Bulkhead Retry con Backoff Exponencial

Tres capas complementarias de SmallRye Fault Tolerance actúan en conjunto. El Circuit Breaker protege de llamadas en cascada: tras N fallos el circuito se abre y el fallback actúa inmediatamente sin llamar al servicio caído. El Bulkhead limita la concurrencia por servicio para que un MS lento no consuma todos los threads del pool. El Retry con backoff exponencial reintenta operaciones transitorias con intervalos crecientes antes de mover a DLQ.

Por qué estos patrones

SmallRye Fault Tolerance implementa MicroProfile Fault Tolerance — las tres capas se configuran con anotaciones (@CircuitBreaker, @Bulkhead, @Retry) sin infraestructura adicional y son composables en el mismo método. La combinación Circuit Breaker + Retry es estándar de industria para servicios que dependen de proveedores externos con fallos intermitentes.

BLOQUE 05
Mensajería y Notificaciones
👨‍🎓 STU Service 📝 ANN Service 🔔 NOT Service
BE-STU-002BE-STU-004 BE-ANN-002BE-NOT-001 BE-NOT-002BE-NOT-003
Publisher / Subscriber Strategy — canales de notificación Factory Method — CDI @Named

El Notification Service consume eventos de RabbitMQ vía Publisher/Subscriber. Internamente aplica Strategy para seleccionar el canal según el campo notification_type del contrato genérico NotificationRequest. Un Factory Method implementado con CDI @Named beans resuelve qué implementación de NotificationStrategy instanciar en runtime, sin condicionales hardcodeados ni redeploy.

Por qué estos patrones

El Strategy desacopla el "qué enviar" del "cómo enviarlo": agregar SMS o Webhook solo requiere una nueva clase que implemente NotificationStrategy sin tocar el core. El Factory con Instance<NotificationStrategy> de CDI es idiomático en Quarkus: busca el bean @Named("EMAIL_HTML") en el contexto sin if-else ni switch. Al registrar un nuevo tipo en BD, el factory lo resuelve automáticamente si el bean CDI existe.

BLOQUE 06
Jerarquía Configurable de Contenido
📁 CNT Service
BE-CNT-001BE-CNT-002BE-CNT-003
Composite Repository con Caché

La jerarquía se modela con Composite: todos los nodos —independientemente de su posición en el árbol— implementan la misma interfaz HierarchyNode. El Content Service no tiene clases Java para "Semestre" o "Clase", solo HierarchyNode. La configuración se carga desde BD en startup y se cachea en memoria. El Repository con caché abstrae ese acceso e invalida la caché cuando cambia la configuración del árbol.

Por qué estos patrones

El Composite es la elección natural para árboles de profundidad variable. Si el administrador agrega un nivel intermedio, cero cambios de código. El nodo raíz y el hoja se identifican por posición en el árbol activo, no por tipo. El Repository con caché evita consultas a BD en cada navegación — el árbol solo se recarga cuando cambia la configuración (cache invalidation por evento).

BLOQUE 07
Asistencia — Contrato Agnóstico al Origen
✅ ATT Service
BE-ATT-001BE-ATT-002 BE-ATT-003BE-ATT-004
Value Object — inmutable State — OPEN → CLOSED Open/Closed Principle

El payload se modela como un Value Object inmutable (AttendanceRecord) que solo valida los campos obligatorios en construcción. Los campos opcionales capture_method y metadata se almacenan tal cual sin interpretación — el servicio es completamente ciego al origen de los datos. El cierre de una clase se modela con State Pattern (transición OPEN → CLOSED), que bloquea cualquier modificación posterior.

Por qué estos patrones

El Value Object garantiza que si un AttendanceRecord se construyó, es válido — la inmutabilidad elimina mutaciones accidentales en el pipeline. El principio Open/Closed en el contrato de API significa que integrar un nuevo origen (biometría, RFID) no requiere abrir el código del servicio: el contrato ya tiene los puntos de extensión (capture_method, metadata). El State para cierre es más expresivo que un booleano: modela que CLOSED es un estado con comportamiento propio.

BLOQUE 08
Reportes Cross-Schema
📊 RPT Service
BE-RPT-001BE-RPT-002 BE-RPT-003BE-RPT-004 BE-RPT-006INF-DB-001
Repository — read-only cross-schema Strategy — formatos de salida Facade

Una Facade (ReportExecutor) orquesta: cargar la definición del reporte, delegar la extracción a Repositories read-only que acceden a los schemas necesarios, y pasar el resultado al Strategy de formato correcto: JsonReportStrategy, CsvReportStrategy o PdfReportStrategy. El acceso cross-schema se garantiza mediante credenciales de BD de solo lectura sobre todos los schemas (ADR-001).

Por qué estos patrones

La Facade simplifica la interfaz al cliente: un solo endpoint, un solo contrato, independiente de cuántos schemas se consulten. El Strategy para formatos desacopla "qué datos extraer" de "cómo renderizarlos" — la misma definición produce JSON, CSV o PDF sin duplicar lógica. Agregar un nuevo formato (ej: XLSX) solo requiere una nueva clase XlsxReportStrategy, sin tocar el Executor ni el contrato.
